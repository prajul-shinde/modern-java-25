Syntax, IO & The Modern Runtime

1.1 The Entry Point (JEP 477)
▼
Warm-ups
  • Type the traditional main and run it: public class Hello { public static void main(String[] args) { System.out.println("hi"); } }
  • Type the modern version: void main() { println("hi"); } — run it
  • Try readln("Enter your name: ") and print the name back
  • Try calling println() vs System.out.println() — confirm both work
Concept Checks
  1. Why does traditional main need the static keyword?
  2. What does void mean as a return type?
  3. What is an unnamed class? When does readln() work?
  4. When should you use traditional main vs modern void main()?
  5. What are the 4 keywords in public static void main(String[] args) and why each one?

1. static is needed because the JVM calls main without creating an instance of the class — it must be callable on the class itself, not on an object.

2. void means the method returns nothing — it does not produce a value back to the caller.

3. An unnamed class is a top-level class with no explicit class declaration (JEP 477). readln() works only inside unnamed classes — it is a preview feature for simple programs.

4. Use modern void main() for quick scripts, learning, and prototyping. Use traditional public static void main(String[] args) for production code, frameworks, and any multi-class project.

5. public — accessible from anywhere (JVM needs to call it); static — called without creating an object; void — returns nothing; main — conventional name JVM looks for; String[] args — holds command-line arguments.

1.2 Module Imports (JEP 476)
▼
Warm-ups
  • Write import java.util.ArrayList; and use it
  • Write import java.util.*; and use ArrayList and HashMap
  • Write import module java.base; and use both — notice the difference
Concept Checks
  1. What is a Java module? Name 3 common platform modules.
  2. What's inside java.base? (Name 4 packages)
  3. When should you use specific imports over module imports?
  4. Why might import module cause problems in production code?

1. A Java module is a named, self-contained unit of code defined via module-info.java. Three common modules: java.base, java.sql, java.desktop.

2. java.base contains: java.lang, java.util, java.io, java.time.

3. Use specific imports in production code for clarity — they make it obvious which exact classes are used and avoid name conflicts.

4. import module is a preview feature (JEP 476) — it may be disabled in production toolchains, and importing an entire module can cause ambiguity if two modules export the same class name.

1.3 Source Launch & JVM Architecture (JEP 458)
▼
Concept Checks
  1. What does java Main.java do that javac Main.java + java Main doesn't?
  2. What is bytecode? Why is it platform-independent?
  3. What is JIT compilation and when does it kick in?
  4. What is the difference between JDK, JRE, and JVM?
  5. What happens inside the JVM: Class Loader → Bytecode Verifier → Execution Engine?

1. java Main.java compiles and runs in a single step without producing a .class file on disk — a convenience for single-file programs.

2. Bytecode is an intermediate binary representation (.class file) that is not machine-specific. Any JVM on any OS interprets the same bytecode — "write once, run anywhere."

3. JIT (Just-In-Time) compilation kicks in at runtime: after interpreting bytecode for a while, the JVM identifies "hot" code and compiles it to native machine code for much faster subsequent execution.

4. JVM — runs bytecode. JRE — JVM + standard libraries needed to run Java. JDK — JRE + compiler (javac) + dev tools; needed to write and compile Java.

5. Class Loader — loads .class files into memory. Bytecode Verifier — checks bytecode is valid and safe. Execution Engine — interprets or JIT-compiles and executes the bytecode.

1.4 Variables: var, Unnamed Variables, Primitives vs Wrappers
▼
Warm-ups
  • Declare one of each 8 primitives. Print them all.
  • Declare Integer a = 127; Integer b = 127; — print a == b. Then try with 200.
  • Use var for a HashMap declaration. Confirm the type is inferred correctly.
  • Use _ in a for-each loop where you only care about the count, not the item.
Concept Checks
  1. List all 8 primitive types and their wrapper equivalents.
  2. What is the Integer cache? What range? Why does it exist?
  3. What is autoboxing? Give an example where it happens automatically.
  4. What is the NullPointerException risk with wrappers? Give a concrete example.
  5. When should you use var? When shouldn't you?
  6. What is an unnamed variable _? Give 2 use cases.
  7. Where does a primitive live (stack or heap)? Where does a wrapper live?

1. byte→Byte, short→Short, int→Integer, long→Long, float→Float, double→Double, boolean→Boolean, char→Character

2. Java caches Integer objects for values -128 to 127. Integer.valueOf(n) within this range returns the same cached object. It exists for performance — small integers are used very frequently.

3. Autoboxing is the automatic conversion of a primitive to its wrapper. Example: List<Integer> list = new ArrayList<>(); list.add(5); — the int 5 is automatically boxed to Integer(5).

4. Integer x = null; int y = x; — causes a NullPointerException because unboxing null tries to call x.intValue() on a null reference.

5. ✅ Use: var list = new ArrayList<String>(); (type obvious from RHS). ❌ Don't: var x = getUserData(); (return type unclear).

6. _ is an unnamed variable. Use 1: for (var _ : list) count++;. Use 2: catch (Exception _) {} (ignore the exception object).

7. Primitives live on the stack (fast, automatic cleanup). Wrappers (objects) live on the heap; the stack holds only a reference.

1.5 Text Blocks, String Immutability, StringBuilder
▼
Warm-ups
  • Create a multi-line JSON string using a text block
  • Use .formatted() on a text block to inject variables
  • Demonstrate string immutability: show s1.concat() doesn't change s1
  • Compare string concatenation in a loop vs StringBuilder — time both
Concept Checks
  1. What are the 3 """ rules for text blocks?
  2. What is the String pool? Difference between "Hello" and new String("Hello")?
  3. Why is String immutable? What happens in memory when you do s += "a" in a loop?
  4. When should you use StringBuilder over String concatenation?
  5. What are the common format specifiers: %s, %d, %f, %.2f, %n, %%?

1. Rule 1: Opening """ must be followed by a newline (content starts next line). Rule 2: Closing """ position determines the indentation stripped. Rule 3: Common leading whitespace is auto-stripped based on closing """ placement.

2. The String pool is a special memory region where string literals are interned. "Hello" — goes into pool, reused. new String("Hello") — always creates a new heap object, bypasses pool.

3. String is backed by a final char[]. Each += creates a brand-new String object on the heap. After 1000 iterations, 1000 objects created — very memory-inefficient.

4. Use StringBuilder when concatenating in a loop or building a string piece by piece — it mutates the same internal buffer, no new objects created.

5. %s=String, %d=integer, %f=float, %.2f=2 decimal places, %n=newline (platform-safe), %%=literal percent sign.

Task 3 String vs StringBuilder Performance ⏱ 15 min [20 marks]
  • →Build a 50,000-character string using + in a loop. Measure time in ms.
  • →Build the same string using StringBuilder. Measure time in ms.
  • →Print result using a text block showing both times and speedup ratio
  • →Add a comment explaining WHY StringBuilder is faster
public class Week1Task3 {
    public static void main(String[] args) {
        int n = 50_000;
        long s1 = System.currentTimeMillis();
        String r1 = "";
        for (int i = 0; i < n; i++) {
            r1 += "a"; // Each += creates NEW String — 50k objects!
        }
        long t1 = System.currentTimeMillis() - s1;

        long s2 = System.currentTimeMillis();
        var sb = new StringBuilder();
        for (int i = 0; i < n; i++) {
            sb.append("a"); // Mutates same buffer — no new objects
        }
        String r2 = sb.toString();
        long t2 = System.currentTimeMillis() - s2;

        double speedup = t2 > 0 ? (double) t1 / t2 : t1;
        System.out.printf("""
            String(+):     %d ms
            StringBuilder: %d ms
            Speedup:       %.1fx%n""", t1, t2, speedup);
    }
}

OOP Pillars I & II: Encapsulation, Records, Inheritance

2.1Encapsulation & Access Modifiers
▼
Warm-ups
  • Write a Person class with public fields. Then make them private. Add getters.
  • Add a setter for age that throws IllegalArgumentException if age < 0
  • Try accessing a private field from outside the class — see the compile error
Concept Checks
  1. What is encapsulation? Why make a field private if you're just going to expose it with a getter?
  2. What are the 4 access modifiers? What can see each one?
  3. What is the this keyword? Give an example of when it's necessary.
  4. What is the difference between a constructor and a method?
  5. Why would you have no setter for a field but still have a getter?

1. Encapsulation bundles data and behavior and controls access. A private field with a getter: you can add validation, make it read-only, change internal representation later without breaking callers — the getter is a stable contract.

2. private — only the same class. package-private (no modifier) — same class + same package. protected — same class + package + subclasses. public — everything everywhere.

3. this refers to the current object instance. Necessary to distinguish field from parameter with same name: this.name = name;

4. A constructor has no return type, same name as the class, called via new. A method has a return type, any name, called on an existing object.

5. A getter with no setter means the field is effectively read-only from outside. Internal logic can still change it — gives external immutability without making the whole class immutable.

2.2Records
▼
Warm-ups
  • Write record Point(double x, double y) {} — create one and call p.x(), p.y(), System.out.println(p)
  • Write two Points with same values — compare with .equals()
  • Add a compact constructor that validates x and y are not negative
  • Add a custom method distance() to the record
Concept Checks
  1. What 5 things does a record auto-generate?
  2. What is a compact constructor? How is it different from a regular constructor?
  3. Can a record extend a class? Can a class extend a record?
  4. Are records mutable or immutable?
  5. When should you use Record vs class? Give 3 examples of each.
  6. What is the difference between person.name() (record accessor) and person.getName() (class getter)?

1. Canonical constructor, accessor methods (one per field, named after the field), equals(), hashCode(), toString().

2. A compact constructor omits the parameter list — it implicitly receives all fields and you write only validation/transformation logic. Fields are assigned automatically at the end.

3. No — a record cannot extend any class (implicitly extends Record). No — a class cannot extend a record (records are implicitly final).

4. Records are immutable — all fields are final, there are no setters.

5. Use Record for: DTO/data transfer, coordinates, API responses. Use class for: mutable objects like BankAccount, objects with complex behavior, JPA entities.

6. Record accessor is person.name() — no "get" prefix. Class getter is person.getName() — with "get" prefix.

2.3Inheritance & super
▼
Concept Checks
  1. What is inheritance? What does extends do?
  2. What are the 3 uses of the super keyword?
  3. What is the order of constructor execution in an inheritance chain?
  4. What is method overriding? What are the 4 rules for overriding?
  5. What does @Override annotation do?
  6. Can you override a final method? A private method? A static method?
  7. Why doesn't Java support multiple class inheritance? What is the diamond problem?
  8. What class does every Java class implicitly extend?

1. Inheritance allows a child class to acquire fields and methods of a parent class. extends establishes the IS-A relationship.

2. super() — call the parent constructor. super.field — access a parent field. super.method() — call a parent method that has been overridden.

3. Top-down: GrandParent constructor → Parent constructor → Child constructor.

4. Overriding = redefining a parent method in child. 4 rules: same name, same parameters, same or covariant return type, not more restrictive access.

5. @Override tells the compiler to verify this is actually overriding a parent method. Without it, a typo creates a new method silently.

6. No — final methods cannot be overridden. No — private methods are not inherited. No — static methods can be "hidden" but not overridden.

7. If class C extends A and B, and both have hello(), the compiler can't know which to use. This ambiguity is the diamond problem. Java avoids it with single class inheritance.

8. Every Java class implicitly extends java.lang.Object.

2.4Sealed Classes
▼
Concept Checks
  1. What problem do sealed classes solve?
  2. What are the 3 options for a permitted subclass: final, sealed, non-sealed?
  3. Why do sealed classes enable exhaustive switch statements?
  4. What is the difference between sealed class and final class?
  5. Can interfaces be sealed?

1. Sealed classes solve uncontrolled inheritance — they give the author control over which classes can extend them, enabling safe exhaustive pattern matching and closed type hierarchies.

2. final — no further subclassing. sealed — can be extended but only by its own permitted list. non-sealed — opens the hierarchy again.

3. The compiler knows the complete, fixed set of subtypes at compile time — it can verify every case is covered in a switch, so no default is needed.

4. final = no subclassing at all. sealed = controlled subclassing — only permitted classes may extend it.

5. Yes, interfaces can be sealed.

Task 1 Encapsulation with Validation — BankAccount ⏱ 15 min [20 marks]
  • →Private double balance — no setter
  • →Constructor sets initial balance (reject if negative)
  • →deposit(double amount) — validates > 0
  • →withdraw(double amount) — validates > 0 and balance sufficient
  • →getBalance() getter + toString()
  • →Test in main: valid deposit, valid withdrawal, invalid amount (catch), overdraft (catch)
public class BankAccount {
    private double balance;

    public BankAccount(double initial) {
        if (initial < 0) throw new IllegalArgumentException("Cannot be negative");
        this.balance = initial;
    }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Deposit must be positive");
        balance += amount;
    }

    public void withdraw(double amount) {
        if (amount <= 0) throw new IllegalArgumentException("Withdrawal must be positive");
        if (amount > balance) throw new IllegalStateException("Insufficient funds");
        balance -= amount;
    }

    public double getBalance() { return balance; }

    public String toString() { return "BankAccount{balance=%.2f}".formatted(balance); }
}

    public Money { // compact constructor
        if (amount < 0) throw new IllegalArgumentException("Amount cannot be negative");
        if (currency == null || currency.isBlank()) throw new IllegalArgumentException("Currency required");
    }

    public Money add(Money other) {
        if (!this.currency.equals(other.currency))
            throw new IllegalArgumentException("Currency mismatch");
        return new Money(this.amount + other.amount, this.currency);
    }

    public String format() { return "%s %.2f".formatted(currency, amount); }

    public static Money zero(String currency) { return new Money(0, currency); }
}

Task 4 Gate Test — Sealed Classes + Records Payment System ⏱ 20 min [25 marks]
  • →sealed interface Payment permits CashPayment, CardPayment, UPIPayment
  • →Each is a record with appropriate fields
  • →Write processPayment(Payment p) using switch pattern matching — no default case needed
  • →Write generateReceipt(Payment p) returning formatted String
  • →In main: create one of each, process all, print receipts
public class PaymentSystem {
    sealed interface Payment permits CashPayment, CardPayment, UPIPayment {}
    record CashPayment(double amount) implements Payment {}
    record CardPayment(String last4digits, double amount) implements Payment {}
    record UPIPayment(String upiId, double amount) implements Payment {}

    static void processPayment(Payment p) {
        // No default needed — compiler knows all permitted types
        switch (p) {
            case CashPayment c  -> System.out.println("Cash: $" + c.amount());
            case CardPayment c  -> System.out.println("Card ****" + c.last4digits());
            case UPIPayment u   -> System.out.println("UPI: " + u.upiId());
        }
    }

    static String generateReceipt(Payment p) {
        return switch (p) {
            case CashPayment c  -> "RECEIPT: Cash $%.2f".formatted(c.amount());
            case CardPayment c  -> "RECEIPT: Card ****%s $%.2f".formatted(c.last4digits(), c.amount());
            case UPIPayment u   -> "RECEIPT: UPI %s $%.2f".formatted(u.upiId(), u.amount());
        };
    }
}
